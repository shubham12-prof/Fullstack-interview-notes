# 12. MFA / 2FA (Multi-Factor / Two-Factor Authentication)

## What is MFA/2FA?

Multi-Factor Authentication requires a user to prove their identity with **more than one** independent factor, drastically reducing the risk of account compromise even if a password is leaked/stolen. 2FA is the common case of exactly two factors.

## The Three Categories of Authentication Factors

| Factor Type            | Examples                                                    |
| ---------------------- | ----------------------------------------------------------- |
| **Something you know** | Password, PIN, security question                            |
| **Something you have** | Phone (SMS/authenticator app), hardware security key, email |
| **Something you are**  | Fingerprint, face recognition (biometrics)                  |

True MFA combines factors from **different** categories — a password plus a security question is technically "two things you know," which is much weaker than password + authenticator app.

## Common MFA Methods

| Method                   | How it Works                                                      | Security Level                                        |
| ------------------------ | ----------------------------------------------------------------- | ----------------------------------------------------- |
| SMS OTP                  | One-time code sent via text message                               | Moderate — vulnerable to SIM-swapping                 |
| Email OTP                | One-time code sent via email                                      | Moderate — as strong as email account security        |
| TOTP (Authenticator App) | Time-based code generated by an app (Google Authenticator, Authy) | Strong — no network dependency, resistant to SIM-swap |
| Hardware Security Key    | Physical device (YubiKey), FIDO2/WebAuthn                         | Strongest — resistant to phishing                     |
| Push Notification        | Approve/deny login via a mobile app prompt                        | Strong, good UX                                       |

## TOTP-Based 2FA (Most Common for Web Apps)

TOTP (Time-based One-Time Password) generates a 6-digit code that changes every 30 seconds, derived from a shared secret and the current time — this is what Google Authenticator, Authy, and similar apps implement.

```bash
npm install speakeasy qrcode
```

### Step 1: Generate a Secret and QR Code (Enrollment)

```js
const speakeasy = require("speakeasy");
const QRCode = require("qrcode");

app.post("/auth/2fa/setup", authenticate, async (req, res) => {
  const secret = speakeasy.generateSecret({
    name: `MyApp (${req.user.email})`, // shown in the authenticator app
  });

  // Store the secret temporarily (not yet enabled) until the user confirms with a valid code
  await User.findByIdAndUpdate(req.user.id, {
    twoFactorTempSecret: secret.base32,
  });

  const qrCodeDataUrl = await QRCode.toDataURL(secret.otpauth_url);

  res.json({
    qrCode: qrCodeDataUrl, // display this as an <img> for the user to scan
    manualEntryKey: secret.base32, // fallback for manual entry
  });
});
```

### Step 2: Verify Setup with a Code from the App

```js
app.post("/auth/2fa/verify-setup", authenticate, async (req, res) => {
  const { code } = req.body;
  const user = await User.findById(req.user.id);

  const isValid = speakeasy.totp.verify({
    secret: user.twoFactorTempSecret,
    encoding: "base32",
    token: code,
    window: 1, // allows a 30s window of clock drift tolerance
  });

  if (!isValid) return res.status(400).json({ error: "Invalid code" });

  user.twoFactorSecret = user.twoFactorTempSecret;
  user.twoFactorTempSecret = undefined;
  user.twoFactorEnabled = true;
  await user.save();

  res.json({ message: "2FA enabled successfully" });
});
```

### Step 3: Requiring the Code at Login

```js
app.post("/auth/login", async (req, res) => {
  const { email, password } = req.body;
  const user = await User.findOne({ email });

  if (!user || !(await bcrypt.compare(password, user.passwordHash))) {
    return res.status(401).json({ error: "Invalid credentials" });
  }

  if (user.twoFactorEnabled) {
    // Issue a short-lived, limited-purpose "pending 2FA" token instead of full access
    const pendingToken = jwt.sign(
      { sub: user.id, purpose: "2fa-pending" },
      process.env.JWT_SECRET,
      { expiresIn: "5m" },
    );
    return res.json({ requires2FA: true, pendingToken });
  }

  const token = generateAccessToken(user);
  res.json({ token });
});

app.post("/auth/2fa/validate", async (req, res) => {
  const { pendingToken, code } = req.body;

  let decoded;
  try {
    decoded = jwt.verify(pendingToken, process.env.JWT_SECRET);
    if (decoded.purpose !== "2fa-pending") throw new Error();
  } catch {
    return res.status(401).json({ error: "Invalid or expired session" });
  }

  const user = await User.findById(decoded.sub);
  const isValid = speakeasy.totp.verify({
    secret: user.twoFactorSecret,
    encoding: "base32",
    token: code,
    window: 1,
  });

  if (!isValid) return res.status(401).json({ error: "Invalid 2FA code" });

  const accessToken = generateAccessToken(user);
  res.json({ token: accessToken });
});
```

> Note the two-step token design: after password verification but before 2FA confirmation, the user holds only a limited "pending" token that can't be used for regular API access — full access is granted only after the second factor succeeds.

## Backup / Recovery Codes

Users can lose their phone/authenticator app — always provide one-time backup codes generated at enrollment, so they aren't permanently locked out.

```js
const crypto = require("crypto");

function generateBackupCodes(count = 10) {
  return Array.from(
    { length: count },
    () => crypto.randomBytes(4).toString("hex"), // e.g., "a1b2c3d4"
  );
}

app.post("/auth/2fa/verify-setup", authenticate, async (req, res) => {
  // ...after verifying the TOTP code as shown above...

  const backupCodes = generateBackupCodes();
  const hashedCodes = await Promise.all(
    backupCodes.map((c) => bcrypt.hash(c, 10)),
  );

  await User.findByIdAndUpdate(req.user.id, { backupCodes: hashedCodes });

  res.json({ message: "2FA enabled", backupCodes }); // show ONCE — user must save these
});
```

```js
// Using a backup code instead of a TOTP code at login
async function tryBackupCode(user, submittedCode) {
  for (let i = 0; i < user.backupCodes.length; i++) {
    if (await bcrypt.compare(submittedCode, user.backupCodes[i])) {
      user.backupCodes.splice(i, 1); // each backup code is single-use
      await user.save();
      return true;
    }
  }
  return false;
}
```

## SMS-Based OTP (Alternative Method)

```bash
npm install twilio
```

```js
const twilio = require("twilio")(
  process.env.TWILIO_SID,
  process.env.TWILIO_AUTH_TOKEN,
);

function generateOtp() {
  return Math.floor(100000 + Math.random() * 900000).toString(); // 6-digit code
}

app.post("/auth/2fa/send-sms", authenticate, async (req, res) => {
  const otp = generateOtp();
  await User.findByIdAndUpdate(req.user.id, {
    smsOtp: otp,
    smsOtpExpires: Date.now() + 5 * 60 * 1000,
  });

  await twilio.messages.create({
    body: `Your verification code is: ${otp}`,
    from: process.env.TWILIO_PHONE_NUMBER,
    to: req.user.phoneNumber,
  });

  res.json({ message: "OTP sent" });
});
```

> SMS-based OTP is convenient but weaker than TOTP due to **SIM-swapping attacks**, where an attacker socially engineers a carrier into transferring a victim's phone number to a new SIM they control. Prefer TOTP or WebAuthn where feasible, especially for high-value accounts.

## WebAuthn / Passkeys (Strongest, Modern Approach)

WebAuthn (via the `@simplewebauthn/server` library) enables phishing-resistant authentication using hardware security keys, platform biometrics (Face ID/Touch ID/Windows Hello), or passkeys — increasingly the recommended direction for both MFA and even passwordless primary authentication.

```bash
npm install @simplewebauthn/server
```

```js
const {
  generateRegistrationOptions,
  verifyRegistrationResponse,
} = require("@simplewebauthn/server");

app.post("/auth/webauthn/register-options", authenticate, async (req, res) => {
  const options = await generateRegistrationOptions({
    rpName: "MyApp",
    rpID: "myapp.com",
    userID: req.user.id,
    userName: req.user.email,
  });
  // store options.challenge temporarily to verify against on the next step
  res.json(options);
});
```

Full WebAuthn implementation involves more setup (challenge storage, credential storage, attestation verification) — covered here at a conceptual level since it's an increasingly important modern authentication direction.

## Common Interview-Style Questions

- **What are the three categories of authentication factors?**
  Something you know (password), something you have (phone/security key), something you are (biometrics). True MFA combines factors from different categories.

- **How does TOTP work at a high level?**
  A shared secret is established between the server and the user's authenticator app during enrollment; both independently compute a time-based code using the secret + current time, and the codes are compared for a match (the server tolerates a small window for clock drift).

- **Why is SMS-based 2FA considered weaker than TOTP or hardware keys?**
  It's vulnerable to SIM-swapping attacks, where an attacker transfers the victim's phone number to a device they control, intercepting the OTP.

- **Why issue a separate "pending" token after password verification but before 2FA confirmation, instead of full access immediately?**
  It ensures the second factor is genuinely required before granting real API access — a pending token has limited/no privileges and can only be used to complete the 2FA challenge, preventing password-only bypass.

- **Why provide backup/recovery codes for 2FA?**
  Users can lose access to their authenticator app or phone; backup codes provide a fallback recovery path so they aren't permanently locked out of their account.
